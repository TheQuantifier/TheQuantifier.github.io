What: Designing Games & Building Computers

Game Design (My Focus)

My main hobby is game design—the process of crafting mechanics, pacing, and feedback into immersive player experiences. I build my own systems and logic structures using Godot 4 and model environments in Blender. My workflow follows a simple but effective loop: concept → prototype → playtest → refine. I document my progress and version control through my public GitHub repository: Isekai Dungeons on GitHub.

  • Genre focus: 3D dungeon RPG with anime-inspired combat
  • Core loop: explore → fight → reward → upgrade → repeat
  • Design pillars: clarity, responsiveness, and progression
  • Version control: Git with LFS for large assets

PC Building (My Focus)

My second hobby—PC building—grew directly out of my passion for game design. I built my own desktop so I could continue developing games without hardware bottlenecks. The process taught me how to optimize thermals, tune BIOS settings, and balance performance for creative workloads. The build itself reflects my design philosophy: functional, efficient, and personal. You can view the full build details on my PCPartPicker page.

  • Build goal: reliable, quiet system for long dev sessions
  • Specs focus: balanced CPU/GPU, 32 GB RAM, NVMe SSD
  • BIOS tuning: XMP/EXPO profiles and custom fan curves
  • Maintenance: regular dusting and thermal repaste

These two hobbies naturally connect—I built my PC to continue pursuing game design. When my laptop could no longer handle heavy 3D scenes or real-time lighting, I saw PC building as the next creative step. Constructing my own system gave me control over performance, silence, and reliability. That hardware foundation now powers every project I design in Godot, turning my technical curiosity into artistic momentum.

When: Timing, Sprints & Burn-in

Overview

I plan my hobby like a lightweight product cycle: set clear milestones, work in short sprints, and protect buffer weeks near exams. Each cycle ships a visible improvement—either a playable slice in Godot or a validated hardware change on my PC.

Design Cadence

Weekly loop: prototype → playtest → adjust scope → polish one thing. I track only 3 goals per week to avoid scope creep and ensure consistent progress toward a vertical slice.

  • Mon/Tue: graybox or mechanic spike
  • Wed: internal playtest & notes
  • Thu: address 1–2 high-impact issues
  • Fri/Sat: polish pass + refactor

Build Timing

I align hardware work with sales windows and school load. I batch changes (BIOS, fans, storage) so each session is measurable and reversible.

  • Sales windows: back-to-school, Black Friday/Cyber Monday, spring refresh
  • Upgrade blocks: 1 evening to install; separate evening for stress tests
  • Quarterly: dust filters, verify temps, quick cable tidy

Playtest Windows

Short, frequent tests beat long, rare ones. I capture 3 hypotheses before each test and log a single success metric (e.g., readable enemy tells, target TTK, or pathing clarity).

  • Session length: 10–20 minutes, twice a week
  • Metrics: FPS frametime stability, input latency feel, player completion time
  • Artifacts: 1 clip or screenshot + 3 bullet takeaways

Burn-in & Maintenance

I stress components separately, then together, and compare against my baseline. Any change that worsens temps/noise gets reverted.

  • CPU: 30–45 min sustained load; watch temps & clocks
  • GPU: real-time 3D scene or stress tool to validate VRAM & frequency
  • Combined: run the actual Godot scene; log average FPS + 1% lows
  • Quarterly: dust filters, visual check of cables, quick paste health check

This is my main workspace for focused sessions in Godot and Blender. I keep the setup quiet and ergonomic so I can iterate prototypes without distraction.

  • Clean desk, single main display (plus laptop as second when needed)

How: I Build the Game & the PC That Runs It

Project Setup & Structure

I structure the Godot project for clarity and iteration speed. Everything is versioned in Git (LFS for large assets) and synced to my public repo.

  • Folders: scenes/, scripts/, assets/, ui/, levels/, tests/
  • Scenes: one scene per feature; keep scripts small and focused
  • Config: per-machine overrides kept out of source control

Repo: Isekai Dungeons on GitHub

Branches, Commits & PRs

I work in short, named branches and keep commits reviewable. Each branch targets one clear change.

  • Branches: feature/, fix/, exp/ (experiments)
  • Commit style: imperative + scope, e.g., feat(combat): add dodge i-frames
  • PR checklist: before/after clip, notes on readability/FPS, test steps

Implementation Pattern

I follow a simple loop to avoid scope creep: spec → stub → graybox → playtest → polish. Code is organized by feature with small, pure functions and minimal global state.

  • Start with a failing test or a tiny scene proving the mechanic
  • Graybox visuals until inputs/feedback feel right
  • Only then add effects, SFX, and UI polish

Testing, Profiling & Benchmarks

I run quick tests after each change and profile weekly to keep performance stable on my rig.

  • Playtests: 10–20 min slices with 3 hypotheses to verify
  • Metrics: frametime stability, 1% lows, input latency feel
  • Tools: Godot profiler, OBS clips, RTSS overlay for FPS/temps
  • Artifacts: short clip + bullet takeaways committed to repo

How We Built the PC (with a Friend)

  1. Trip to Micro Center (Charlotte): compared parts in person, confirmed stock, and matched CPU/GPU to my Godot workload.
  2. Buy the parts: CPU, motherboard, RAM, GPU, PSU, case, NVMe, fans, thermal paste.
  3. Flat-table assembly: installed standoffs, mounted CPU + cooler, seated RAM/NVMe, placed board in case, routed PSU cables.
  4. First power-on (POST): checked for display, fan spin, and BIOS detection; fixed any loose connections.
  5. Move to my room & setup: cable management, monitor/keyboard/mouse, network, and a quiet fan curve.
  6. Install OS & drivers: Windows/Linux, chipset/GPU drivers, updates; enabled XMP/EXPO.
  7. Test & validate: stress CPU/GPU, verify temps/noise, run a Godot scene and log FPS + 1% lows.
